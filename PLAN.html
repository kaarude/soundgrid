Project Plan · v0.1

Soundboard

An open-source, cross-platform soundboard app (Windows-first) that plays audio clips directly into your active microphone input — so games, Discord, OBS, and any voice app hear your sounds as if they came from your mic. No extra hardware. Distributed as a single .exe.

1 Everything We Need

A breakdown of the building blocks required to ship this app for Windows first, then macOS.

Core components

  • GUI app shell — the main window with a grid of sound buttons, a sidebar for categories, a search bar, and a settings panel.
  • Audio engine — loads, decodes, and plays audio files. Handles volume, looping, hotkeys, and overlapping playback (so a new clip can cut off or layer on the previous one).
  • Mic injection layer — the hard part. Plays audio into the microphone stream so other apps receive it as mic input. Requires a virtual audio device (see §5).
  • Sound library / storage — a local folder (e.g. %APPDATA%/Soundboard/sounds) where imported files live. Plus a library.json index with names, categories, hotkeys, and volume per clip.
  • Hotkey system — global keyboard shortcuts that work even when the app is minimized (system-wide hotkeys), so you can fire a sound mid-game.
  • Settings persistence — saves config (selected mic, output volume, theme, hotkeys) to a JSON file in app data.
  • System tray integration — runs in the background, quick access to mute, stop-all, and open the window.
  • Installer / packaging — produces a signed .exe installer (or portable exe) for Windows; later a .dmg for macOS.

Suggested tech stack (open-source friendly)

Electron + TypeScript
Cross-platform UI, easy packaging to .exe/.dmg. Large ecosystem.
Tauri + Rust + TS
Smaller bundle, lower RAM, modern. Great for a lightweight .exe.
.NET / WPF (C#)
Native Windows-first option, excellent audio & hotkey APIs. Hardest to port to macOS.
Virtual Audio Cable (VB-CABLE)
Free virtual mic driver we can bundle/recommend for mic injection.
Recommended path: Tauri (or Electron) + TypeScript for the cross-platform UI, with a small native helper for the audio/mic-injection on Windows. Recommend VB-CABLE as the virtual mic device on first run, and document a one-click setup. This keeps it open-source, scriptable, and shippable as a single .exe.

3 Supported Audio Files

Decide what formats the library accepts and what gets converted internally.

Supported on import

  • MP3 Most common, small, great for clips.
  • WAV Lossless, instant seek, zero decode overhead — best for short SFX.
  • OGG / Vorbis Open format, good compression.
  • FLAC Lossless, larger files — supported but not necessary.
  • M4A / AAC Common on macOS / iTunes libraries.
  • OPUS Nice-to-have, great quality per byte.

Recommendation

Accept MP3, WAV, OGG, M4A/AAC, FLAC on import. Internally decode with a single audio backend (e.g. FFmpeg / miniaudio / Howler) so every format Just Works. Optionally normalize loud clips to a target peak so the mic output doesn't clip.

Not recommended Avoid streaming-only / DRM formats and raw containerless bitstreams. No video files — if a user drops an .mp4, extract and import the audio track only.

4 Important to Know

  • Windows mic injection is not built into the OS. You cannot simply "play to the microphone." You need a virtual audio device that acts as a fake mic. Apps like Soundpad ship their own signed kernel driver; an open-source project should rely on a free virtual cable (VB-CABLE) or build its own WDM driver (much harder, requires signing).
  • Driver signing. On Windows 10/11 x64, kernel drivers must be WHQL-signed or run with test-signing on. This is why bundling a third-party free cable is the pragmatic open-source route.
  • You still need to mix your real mic in. If you route sounds to the virtual cable, apps won't hear your real voice unless the app mixes real-mic + cable, or our app does the mixing. So we should capture the real mic and mix it with the sounds before sending to the virtual device.
  • Latency matters. For gaming/Discord use, keep round-trip latency under ~50ms. Use WASAPI exclusive or low-buffer shared mode on Windows.
  • Global hotkeys require OS hooks. On Windows this is RegisterHotKey / low-level keyboard hook; on macOS it needs Accessibility permissions.
  • Privacy & permissions. Recording/mixing the real microphone triggers OS microphone privacy prompts on both Windows and macOS — the app must declare and request mic access.
  • Single .exe expectation. Bundle the app + audio engine into one installer. The virtual cable driver is the one external piece — ship a guided installer for it, or auto-detect and prompt.
  • Open-source license. Pick a permissive license (MIT or GPLv3). If we depend on VB-CABLE, note that VB-CABLE itself is donationware, not strictly OSS — so long-term we may want our own driver or a truly OSS virtual device.
  • Distribution size. Electron ≈ 80–150MB; Tauri ≈ 5–15MB. For a "download and run" .exe, Tauri is dramatically leaner.

5 Pause / Player — Microphone Only

A dedicated "mic player" whose job is only to feed the microphone stream. It is independent of what you hear.

  • Separate play/pause for the mic stream. The mic player can be paused without affecting your local monitor. So you can keep listening to a clip in your headphones while it is not being broadcast to others.
  • Mic pause button — a global toggle (and a hotkey) that mutes/pauses everything going to the mic while leaving local playback running. Useful for "let me preview this before I send it."
  • Mic mute toggle — hard mute of the mic output (the virtual device receives silence) while your real voice (if mixed) still works. Different from pausing a clip: this mutes the whole mic channel.
  • Independent transport. Play / Pause / Stop / Loop controls for the mic output, separate from the headphone monitor. Two transport bars in the UI: Mic out and Headphones.
  • Per-clip mic enable. A clip can be marked "monitor only" (heard by you, not sent to mic) or "broadcast" (sent to mic). Great for testing clips before going live.
In short: the mic player is its own pipeline with its own play/pause/mute, fully decoupled from your headphone listening. You control what others hear independently from what you hear.

6 Headset Routing — Mic Out & Headphone Out Only

Strict two-output model: one stream goes to the microphone, one stream goes to your headphones. Nothing else.

Routing model

  • Mic output → the selected virtual audio device (what apps see as your microphone). Only this output is heard by other people.
  • Headphone output → your headset / default playback device. Only this output is heard by you.
  • No third output. The app never sends audio to system speakers, default audio out, or any other device. Just the two named outputs.
  • Headset-only mode. When enabled, monitor audio is forced to the currently selected headset and never leaks to speakers — useful on systems where the default device is desktop speakers.
  • Mic-only mode. When enabled, the clip is sent only to the mic output and you hear nothing locally (pure "broadcast a sound" mode).

Signal flow

Sound clip
   │
   ├──► [Mic bus]   ──► volume ──► Virtual Audio Cable  ──► apps hear it as mic
   │                                  (+ mixed with your real mic if passthrough ON)
   │
   └──► [Monitor bus] ──► volume ──► Headset / headphones  ──► only YOU hear it
Two buses, two destinations, two independent volumes and play/pause states. The mic bus never touches the headphone device, and the monitor bus never touches the mic device. Exactly the Soundpad behavior, made explicit.
